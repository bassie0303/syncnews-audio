# バックログ（後日対応の候補）

直近で着手しない改善アイデアの置き場。優先度・粒度は随時見直す。

## UX / プレーヤー
- [ ] **表示/音声プリセット4種の1タップ切替**
  「表示言語 × 音声言語」の組み合わせを4プリセットとして1タップで巡回切替できるようにする:
  1. 日本語表示 × 日本語音声
  2. 英語表示 × 英語音声
  3. 日本語表示 × 英語音声
  4. 英語表示 × 日本語音声
  現状はテキスト/音声を独立トグル（2スイッチ）。これに加えて「よく使う4組」を素早く回せるUIを用意する。
  例: プリセットチップの横スクロール or 1ボタンで 1→2→3→4 を巡回。

- [ ] **記事ごとの再生形式（4パターン）選択 + 設定の引き継ぎ**
  - 各記事に対して、上記4パターンのどれで再生/表示するかを選べる。
  - **次に再生・表示する記事は、直前に選んだ設定を引き継ぐ**（最後に使ったプリセットが既定になる）。
  - 実装メモ: 「現在の選好プリセット」をアプリ状態として保持（永続化も検討）。プレーヤーを開くとき
    その選好を初期値に。記事個別に上書きした場合の保持範囲（その記事だけ／以降の既定も更新）を要設計。
    上の「プリセット4種の1タップ切替」と一体で考える。

- [ ] **最初に戻るボタン** — 再生位置を記事の先頭（最初のセグメント）に一発で戻すボタン。プレーヤーのコントロールに追加（`audio.seek(Duration.zero)` ＋ index リセット）。

- [ ] **リピート機能** — 記事全体の繰り返し再生。将来的には「文単位リピート（今の文を数回）」「区間リピート（A-B）」も検討。まずは記事末→先頭へ自動ループのトグルから。

- [ ] **スリープタイマー機能** — 「15分後／30分後／再生中の記事の最後まで」などで自動停止。就寝前のリスニング用。タイマー残り時間の表示と、終了時のフェードアウトも検討。

## 音声品質
- [ ] **日本語の漢字読み精度の改善**
  ElevenLabs（多言語モデル）は漢字の読み・固有名詞・数字を誤読しがち。対策候補:
  - 日本語トラックを日本語特化TTS（Google Cloud TTS / Azure / VOICEVOX 等、タイムスタンプ取得可能なもの）に出し分け
  - もしくは形態素解析で読み（カナ）に変換してから合成（※表示は漢字のまま・文単位で対応付け）
  - ElevenLabs 発音辞書で頻出固有名詞を登録
  ※ 本項目は別途着手中（形態素解析→カナ化アプローチ）。

- [ ] **日本語音声の不自然な区切り（音引き・単語途中での分割）** — 長音符「ー」が区切り扱いになっている疑い。その他、単語の途中で音声が切れる箇所がありそう。
  調査対象: カナ化（`_to_reading_ja`）の分かち書き、文分割（`_split_sentences`/`aggregate_to_sentences`）の境界規則、ElevenLabsの読み上げ単位。録音とセグメント境界（start_ms/end_ms）を突き合わせて、どの段で切れているか切り分ける。

## 同期精度
- [ ] **日↔英の文数1:1保証**
  翻訳がGPTで文を統合/分割すると日英の文数がズレ（例 ja32/en44）、言語切替時の `mapPositionForLanguageSwitch` が相対進捗フォールバックになる。文単位翻訳（原文を文分割してから各文を翻訳）で1:1を保証する。

## 記事管理 / 一覧
- [ ] **履歴（記事）の削除** — 一覧から記事を削除（DB行＋Storage音声も合わせて削除）。スワイプ削除 or 長押しメニュー。
- [ ] **コンバートのキャンセル** — 変換中（processing）の記事を中断できるようにする。バックエンド側のジョブ中断＋status遷移（cancelled 等）の設計が必要。
- [ ] **コンバート開始（受付）の明確化** — URL登録時に「受け付けました／変換中」を分かりやすくフィードバック（トースト/一覧の即時pending表示/プログレス表現）。現状はスナックバーのみ。
- [x] **記事の掲載日付・時刻の表示** — 抽出時に生HTMLの構造化データ（JSON-LD `datePublished` / `<meta article:published_time>` 等 / `<time datetime>`）から公開日時を取得し `articles.published_at` に保存、一覧カードとプレーヤーAppBarに表示。
  - DDL: `supabase/migration_published_at.sql`（`alter table articles add column if not exists published_at timestamptz;`）を本番に適用必須。
- [ ] **複数デバイス間で中断地点を引き継ぐ** — ある端末で途中まで聴いた記事を、別端末で続きから再生できるようにする。
  再生位置（記事ID＋言語＋position_ms＋更新時刻）をサーバ（Supabase）に保存し、開いたとき最新の位置から再開。
  実装メモ: `playback_progress` テーブル（または articles に列追加）＋ 定期/一時停止時の保存。Phase2のユーザー認証が入ると「誰の進捗か」を厳密化できる（MVPは端末横断の単一ユーザー前提）。

## 既知の不具合
- [ ] **一覧がアプリ復帰時に空になることがある** — BACKでバックグラウンド化後など、Realtimeストリームの再取得が走らず一覧が空表示になる場合がある。`resume` 時の再購読 or プルリフレッシュで対処。

### 再生の安定性（要調査・関連しうる一群）
以下は同根の可能性あり（音声出力/フォーカス/バックグラウンドサービス周り）。再現条件のログ取りから着手する。
- [ ] **英語音声が極端に小さくなることがある** — 特定の再生で英語トラックの音量だけ極端に下がる。原因候補: 生成音声（ElevenLabs）側のラウドネス差、トラック切替時の音量設定、OSのメディア音量/ダッキング。日英のラウドネス正規化（生成時 or 再生時のゲイン調整）を検討。
- [ ] **音が出ないことがある（起動時・再生開始時・途中で消える）** — オーディオセッションの初期化タイミング、`just_audio` の音源ロード/フォーカス取得失敗、音声フォーカスの喪失（他アプリ/通知）などが疑わしい。再現時のログ（playbackState/フォーカス変化）を収集して切り分ける。
- [ ] **バックグラウンド再生が停止することがある** — 上記「音が出ない」と一環の可能性。フォアグラウンドサービスの維持、OSの省電力（Doze/バッテリー最適化）によるkill、オーディオフォーカス喪失などを確認。`docs/native_setup.md` の設定とあわせて点検。
